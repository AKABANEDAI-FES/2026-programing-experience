# 学祭展示「プログラミング体験会」フロントエンド要件定義

## 1. アプリケーションの目的と対象画面
* **対象端末:** 受付・制作PC（計8台）および標準的なマウス
* **目的:** 来場者が迷わず「1画面1タスク」でキャラクターを描き、動きをプログラミングして大画面へ放流する。
* **画面遷移の基本ルール:** お絵かきからプログラミングへは**「完全一方通行」**とし、前の画面に戻るボタンは配置しない（体験時間の管理は近くのスタッフがアナログで行うため、システム側でのタイマーは設けない）。

## 2. 状態管理（State）の定義
*React内で保持し、画面間で引き回すデータの定義。*

| State名 | 型 | 役割 |
| :--- | :--- | :--- |
| `step` | `number` (1〜4) | 現在の画面位置を制御する。 |
| `drawMode` | `'free' \| 'preset' \| null` | 選択されたモード（白紙か、既存キャラか）。 |
| `imageData` | `string` | SVGまたはBase64等で保持される最終的な画像データ。 |
| `program` | `string[]` | ユーザーが選択した動き・セリフのコマンドIDリスト。 |

## 3. 各画面のUI要件・機能要件

### 【画面①】モード選択画面
* **機能:** 以下の2つの入り口（ボタン）を配置する。
  1. **「ゼロから自由にお絵かき」**: 白紙キャンバスからスタートする（画面②へ遷移）。
  2. **「イニャーでいきなりスクラッチ」**: 既存のキャラクター画像を使用し、お絵かきをスキップする（`imageData`に初期画像をセットし、直接画面③へ遷移）。

### 【画面②】お絵かき画面（※Canvas APIは非推奨）
* **機能:**
  * マウスドラッグで線を描き、消しゴムで消す軽量なペイント機能（SVGのPath要素などを利用し、Canvas機能は使わない方針で実装）。
* **遷移トリガー:** 「次へ」押下時、描画状態をデータ化して `imageData` に保存し、画面③へ進む（戻る操作は不可）。

### 【画面③】スクラッチ（プログラミング）画面
* **機能:**
  * **ドラッグ＆ドロップ（D&D）**による直感的なプログラミングUI。
  * 利用可能なコマンドとして「ループさせる動きブロック」や「セリフ（吹き出し）ブロック」を用意。
  * 追加できるコマンドの数は最大5個まで。
* **遷移トリガー:** 「放流」押下時、全データをJSONにまとめ、APIへPOST送信。成功後、画面④へ進む（戻る操作は不可）。

### 【画面④】完了画面
* **機能:** 成功メッセージの表示。5秒経過後、自動的にStateを初期化し画面①へ戻る。

## 4. 大画面モニター（展示用画面）要件
* **目的:** 参加者が放流したキャラクターを受け取り、指定されたプログラムに従って画面内を泳がせる。
* **表示数の上限と管理:** 画面のパフォーマンス低下を防ぐため、**生き物の最大数は「30匹」**とする。
  * 31匹目が放流された場合、最も古いキャラクターを配列から削除する（FIFO: First In, First Out 方式）。
* **通信仕様:** バックエンドからのプッシュ通知（WebSocket等）を受け取り、リアルタイムに画面へ反映させる。

## 5. 今後追加が必要な機能・セーフティ要件（展示用暴走防止）
本番展示に向け、以下の機能を実装する。

* **【高優先】セーフティ機能（受付PC用キオスク設定）:** * 右クリックメニューの禁止。
  * F5（リロード）やF12（開発者ツール）などのキーボード入力の無効化処理。
* **【低優先】管理者画面の追加:**
  * アプリの強制リセット（最初に戻る機能）などを備えた、スタッフ用の隠し画面の検討。
