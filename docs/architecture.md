# システムアーキテクチャ・データフロー

## 1. コンポーネント
このシステムは主に以下の4つの要素で構成されます。

- **参加者画面（フロントエンド：React）**: 子どもがお絵かきやプログラミングをする端末。
- **APIサーバー（バックエンド：Cloudflare Workers）**: データを受け取り、各所へ振り分ける関所。
- **ストレージ（Cloudflare R2）**: 画像データを安全に保存するクラウドの保管庫。
- **大画面モニター（展示用フロントエンド）**: 会場でみんなの描いた絵が泳ぐメイン画面。

## 2. データの流れ
参加者がタブレットで「放流する」ボタンを押してから、大画面に絵が表示されるまでの流れは以下の通りです。

1. **参加者画面**が、絵のデータ（画像）とプログラム（JSONデータ）を**APIサーバー**に送信します。
2. **APIサーバー**は、受け取った画像データを**ストレージ**に保存します。
3. 同時に**APIサーバー**は、リアルタイム通信（WebSocket等）の仕組みを使い、**大画面モニター**へ「新しい絵が追加されたよ！」という通知を即座にプッシュ送信します。
4. 通知を受け取った**大画面モニター**が、**ストレージ**からその絵とプログラムを読み込み、画面の海へと泳がせます。

---

## 3. リアルタイム通信を採用する理由
大画面モニター側から「新しい絵は来ましたか？」と一定時間ごとに何度も確認しに行く方式（ポーリング）ではなく、サーバーからのプッシュ通知（リアルタイム通信）を採用した理由は以下の通りです。

- **無駄な通信（リクエスト）の削減**: 参加者が誰も描いていない時間帯でも、大画面が問い合わせを続けるのは非効率なため。
- **サーバー負荷の軽減**: 大量のリクエストによるCloudflareの利用制限や、システム全体のパフォーマンス低下（遅延など）を防ぎ、安定した展示運営を行うため。
